# LLM 高并发、排队与调度说明

## 1. 先说结论

当外部一下子来了 `10000` 个请求，但某组大模型实例稳定活跃并发只能做到 `100` 时，系统不是“同时硬吃下 10000 个请求”，而是通过以下机制把流量变成可控负载：

1. 接入层先接住请求
2. 做准入控制、限流、优先级划分
3. 能立即返回的请求优先走缓存、小模型或简单链路
4. 真正需要大模型的请求进入有界队列
5. 调度器始终只放最多 `100` 个活跃请求进入推理层
6. 执行结束后再补位
7. 高峰期触发降级、熔断或拒绝部分请求

本质上，这是一套 **接入控制 + 排队系统 + 调度系统 + 路由分流 + 批处理 + 降级策略**，而不是某个 Agent 框架单独“扛住了百万并发”。

## 2. `Eino` 在这里扮演什么角色

`Eino` 更像编排层：

- 构造 prompt
- 调模型
- 调工具
- 串联工作流 / Agent

因此决定系统上限的关键通常不是 `Eino` 本身，而是：

- 应用层入口 QPS
- 单请求会放大成多少次 LLM 调用
- 下游模型推理层 token 吞吐
- 工具服务和检索服务的容量

对于 Agent 模式尤其要注意“请求放大”。一个用户请求可能会变成：

- `N` 次模型调用
- `M` 次工具调用

这意味着入口 `1000 QPS` 不等于下游只有 `1000 QPS`。

## 3. 为什么聊天产品看起来能扛很大流量

“豆包、元宝这类产品能扛海量请求”并不是因为它们让所有请求都直达同一个大模型，而是因为它们通常有分层架构：

### 3.1 接入层

- API Gateway / 负载均衡
- 鉴权
- 限流
- 熔断
- 超时控制

### 3.2 业务编排层

- 会话管理
- Prompt 组装
- RAG / Tool / Agent 路由
- 上下文裁剪

### 3.3 模型路由层

- 简单问题走小模型
- 热门问题走缓存
- 检索问答优先走 RAG
- 复杂任务再走大模型

### 3.4 推理层

- Continuous batching
- KV cache / prefix cache
- GPU 调度
- 多副本扩容
- 长短请求调度

### 3.5 降级层

- 减少输出 token
- 关闭复杂 Agent 链路
- 拒绝低优先级请求
- 切换到更快更便宜的模型

## 4. “10000 请求进来，但模型活跃并发 100”到底怎么处理

可以把它想成银行柜台：

- 来了 `10000` 个客户
- 只有 `100` 个窗口能稳定处理业务
- 于是 `100` 个先进去，其余排队
- 柜台空出位置后，队列中的下一个补位

但实际系统不会只做“死排队”，因为那样等待时间会过长。真正会组合使用：

1. **限流**：超过接入上限的请求直接拒绝或稍后重试
2. **优先级队列**：付费用户、核心链路优先
3. **分流**：并非所有请求都走大模型
4. **缓存**：热门问题直接返回
5. **批处理**：多个请求动态拼 batch 提升 GPU 利用率
6. **取消**：用户断开、超时、主动停止时释放资源

## 5. Go / 后端实现思路

下面用后端工程视角解释如何实现这件事。

### 5.1 核心原则：只让可控数量的请求进入推理层

应用服务不要在收到请求后立刻无脑下发到模型。更常见的做法是：

- HTTP 层接收请求
- 判断是否可直接返回缓存
- 判断是否需要走小模型 / 轻链路
- 剩余请求进入有界等待队列
- 调度器按照配额把请求送入模型层

最重要的是 **bounded queue（有界队列）**。如果队列无上限，高峰期系统会把自己拖死：

- 内存上涨
- 等待时间过长
- 下游超时堆积
- 最终雪崩

### 5.2 一个典型的 Go 分层

可以拆成下面几层：

1. `handler`：解析请求、返回流式结果、处理取消
2. `admission controller`：判断当前是否还能接单
3. `router`：缓存 / RAG / 小模型 / 大模型分流
4. `queue manager`：维护有界队列和优先级队列
5. `scheduler`：控制进入模型层的活跃请求数
6. `llm client`：调用真正模型服务
7. `metrics`：打点等待时间、执行时间、拒绝率、取消率

### 5.3 队列设计

队列至少要记录：

- 请求 ID
- 用户 / 租户 ID
- 优先级
- 入队时间
- 超时时间
- 请求上下文
- 返回通道 / stream writer

工程上需要重点控制：

- 最大队列长度
- 单用户最大排队数
- 单租户最大并发数
- 最大等待时长

如果达到阈值，宁可返回 `429` 或“系统繁忙”，也不要让请求无限排队。

### 5.4 调度器设计

调度器负责维护一个“活跃槽位数”，例如 `100`：

- 当前活跃数 `< 100`：从队列取任务投入执行
- 当前活跃数 `>= 100`：新任务继续排队
- 某任务完成 / 取消 / 超时：释放一个槽位，再补一个

这和 Go 中常见的 **worker pool + semaphore** 很像，但对 LLM 更建议把“真正占模型资源”的阶段限制住，而不是简单限制 HTTP goroutine 数量。

### 5.5 Streaming 的处理

聊天产品通常使用流式返回，Go 侧常见做法是：

- HTTP handler 持有 `context.Context`
- 下游模型 token 到达后通过 channel 往上游写
- 客户端断开时 `ctx.Done()` 触发
- 立刻通知调度器取消对应任务

这一步非常关键，因为很多请求并不会正常结束，而是中途被用户关闭。若不及时取消，会白白占着模型槽位。

### 5.6 超时与取消

至少分三类超时：

- 入队等待超时
- 模型执行超时
- 工具 / 检索子调用超时

还要支持：

- 用户主动停止生成
- 网关断连
- 上游取消整个会话

取消之后要做到：

1. 从队列中移除未执行任务
2. 若任务已在运行，则取消下游调用
3. 释放并发槽位
4. 记录取消原因

### 5.7 高峰降级

当队列过长或延迟过高时，Go 服务需要有显式降级策略：

- 减少最大输出 token
- 关闭 Agent 多步推理，改走单轮回答
- 关闭昂贵工具调用
- 优先走缓存与检索
- 把复杂请求转为异步任务

对于 `stock_rag` 这类场景，复杂多步 Agent 应该只在必要时触发。

### 5.8 为什么不能只靠 goroutine“硬顶”

Go 非常擅长扛高并发连接，但这不等于它可以直接解决大模型容量问题。

原因是：

- goroutine 成本低，但 GPU 资源昂贵
- HTTP 并发高，不代表模型 token 吞吐高
- LLM 真正瓶颈在推理层和显存

所以 Go 层主要负责的是：

- 接住流量
- 管理排队
- 做好分流
- 控制进入模型层的速率

## 6. 对当前 `stock_rag` 项目的启发

你当前项目中有 Agent / ReAct / Plan-and-Execute 的设计，这意味着天然存在“多轮放大”的风险。比较实用的建议是：

1. 默认请求先走简单 RAG / 单轮生成
2. 只有复杂分析任务才进入 Agent 模式
3. 严格限制 `MaxSteps`
4. 给工具调用设置超时与熔断
5. 给模型调用增加并发配额和等待队列
6. 优先缓存股票基础信息、财务数据和热门问题结果

## 7. 一个最实用的容量思维模型

以后评估系统时，不要只问“能不能百万并发”，而要问：

1. 入口峰值 QPS 是多少
2. 平均每个请求触发几次 LLM 调用
3. 平均输入 / 输出 token 是多少
4. 模型层最大活跃并发是多少
5. 最大等待队列长度是多少
6. 超时 / 拒绝 / 降级阈值是什么
7. 命中缓存和小模型分流的比例是多少

只有这些指标清楚了，系统容量才算真正可估。

## 8. 一句话总结

当 `10000` 个请求同时到来，而大模型稳定活跃并发只能处理 `100` 时，正确做法不是让 `10000` 个请求一起冲进模型，而是通过 **准入控制、排队、优先级调度、路由分流、批处理、缓存、取消、扩容与降级** 把请求变成模型层能够稳定消化的负载。
