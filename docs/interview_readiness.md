## stock_rag 面试完善建议

本文档从“AI 相关岗位面试项目”的视角，整理 `stock_rag` 当前的亮点、短板和后续完善优先级。

### 一、当前项目适合拿去面试吗

可以，但更适合作为 **AI 应用工程 / RAG 工程 / Agent 工程 / Go 后端 + AI** 方向的项目骨架，而不是已经完全打磨好的“强展示型项目”。

当前已有的加分点：

- 清晰的工程分层：`api / service / eino / repository / vectorstore`
- 提供 `RAG` 同步查询与流式查询接口
- 接入了 `Eino`，具备基础链路编排能力
- 有统一 `LLMClient`、并发控制、队列调度与统计接口
- 有一定测试覆盖，`go test ./...` 可通过
- 有并发队列设计文档，适合展开讲系统设计

### 二、当前项目最容易被面试官追问出来的问题

#### 1. RAG 主链路还不够“真”

当前检索链路支持：

- 向量检索
- 文档仓库匹配
- 本地样本回退
- fallback 占位结果

问题在于：如果面试官继续问“真实检索效果如何、真实数据是否完整打通”，目前答案还不够硬。

#### 2. Agent 更像单轮包装，不像完整多步 Agent

当前 Agent 主要是：

- 把工具描述写进 prompt
- 发起一次模型调用
- 返回结果

但还没有完整的：

- action 规划
- tool 选择
- tool 参数生成
- tool 执行
- observation 回填
- 多步循环直到完成

因此它更像“Agent 骨架”，不算成熟的工具调用 Agent。

#### 3. 评估体系不足

目前项目能跑，但还缺少：

- 检索命中评估
- 回答 groundedness 评估
- citation 准确性评估
- 不同检索方案的对比实验

这会导致项目很难回答“你怎么证明效果更好”。

#### 4. 工程化还有 demo 痕迹

例如：

- 部分 fallback / skeleton 逻辑较重
- 日志方式仍偏调试式
- README 中对真实演示链路、评估结果、架构图展示还不够完整

### 三、面试前最值得优先补的内容

#### P0：优先级最高，建议先补

1. **打通真实 RAG 闭环**
   - 导入真实股票文档
   - chunk 切分
   - embedding 生成
   - 写入向量库
   - 按 `stock_code / doc_type / time_range` 检索
   - 返回带 citation 的答案

2. **把 Agent 改成真实 tool-calling loop**
   - LLM 输出 action
   - 解析 action 选择工具
   - 执行工具
   - 将 observation 回填上下文
   - 多轮直到得到 final answer 或达到 `MaxSteps`

3. **补最小评估集**
   - 准备 20~30 条股票投研问题
   - 比较 no-RAG、关键词检索、向量检索、hybrid retrieval
   - 记录 hit rate、citation 质量、平均延迟

4. **完善 README 与演示材料**
   - 增加架构图
   - 增加检索流程图
   - 增加 demo 请求/响应示例
   - 增加评估结果表格

#### P1：强烈建议补

5. 检查并修正 `pgvector` 的建表、upsert、过滤与查询实现细节
6. 增加可观测性：请求数、耗时、队列等待、超时、取消、拒绝等指标
7. 增加结构化日志与 `request_id` 贯穿
8. 补齐 `agent / vectorstore / chatmodel` 相关测试

#### P2：有时间再补

9. 增加简单前端或命令行 demo
10. 加缓存、限流、降级策略
11. 支持多模型路由
12. 支持多轮对话记忆

### 四、不同岗位应该突出什么

#### 1. AI 应用 / Agent / RAG 工程岗

重点突出：

- 真 RAG 闭环
- Agent 工具调用
- citation
- stream 输出
- 效果评估

#### 2. AI 平台 / 推理网关 / LLM Infra 岗

重点突出：

- 统一 `LLMClient`
- 有界队列
- 并发控制
- timeout / cancel
- 拒绝与降级策略
- 指标与调度设计

#### 3. 偏算法 / 模型方向岗位

当前项目还不够，需要额外补：

- embedding 模型对比
- rerank 实验
- prompt 策略 ablation
- 更正式的评测报告

### 五、建议你在面试里怎么讲这个项目

可以按下面的结构来讲：

1. **业务背景**：股票投研场景中，公告、研报、新闻分散，人工检索成本高
2. **项目目标**：构建一个支持检索增强、引用返回、流式输出和 Agent 分析的投研问答系统
3. **系统设计**：Go 服务分层 + Eino 编排 + 向量检索 + LLMClient 队列调度
4. **工程亮点**：统一模型调用、并发控制、流式输出、fallback 保证本地开发稳定
5. **当前不足与优化**：继续完善真实检索闭环、Agent 多步工具调用和评估体系

这样的讲法会让项目更像“持续演进的工程系统”，而不是一次性的 demo。

### 六、如果只有 3~5 天准备面试，推荐冲刺顺序

#### 第 1 天

- 彻底打通真实 RAG 查询链路
- 自测导入、检索、回答、citation 是否完整可演示

#### 第 2 天

- 改造 Agent 为真实多步 tool-calling
- 补一组可演示的任务样例

#### 第 3 天

- 做 20~30 条评测问题
- 输出简单评测结果表
- 完善 README、架构图、流程图

#### 如果还有时间

- 补 metrics / tracing
- 修正向量存储实现细节
- 增加一个前端 demo 页或脚本化演示

### 七、最后的判断

一句话总结：

> `stock_rag` 现在已经有“像 AI 工程项目”的骨架，但要想在面试里更有说服力，最关键的不是继续堆接口，而是把“真实检索 + 真实 Agent + 评估 + 可演示性”补齐。

如果这四块补起来，这个项目在 AI 应用工程面试中的说服力会明显提升。
