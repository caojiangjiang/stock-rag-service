后半段怎么讲优化
然后你可以顺着说：

对我来说，准确率低不是一句“模型效果不好”就结束了，而是要继续往下拆。
我一般会把问题拆成四层来看。

1. 先做误差分析，而不是盲目调参
我会先抽样分析错误 case，判断问题到底出在哪一层：

是召回问题，相关文档根本没被检索出来；
是排序问题，正确文档召回了，但排位太靠后；
是生成问题，上下文已经给了，但模型还是总结错了或者产生幻觉；
还是评测问题，比如标注标准不一致、答案本身有歧义，导致准确率被低估。
只有先把错误类型分布搞清楚，后面的优化才有针对性。

2. 优先优化检索链路
在 RAG 这类项目里，我通常会优先看检索，因为很多生成错误，本质上是“输入给错了”。
具体会做几件事：

优化 chunk size 和 overlap，避免关键信息被切碎；
尝试 BM25 + 向量检索 的混合召回，而不是只靠 embedding；
引入 reranker 做二阶段排序，提高前几条上下文的相关性；
调整 top-k，看上下文过少导致信息不足，还是过多导致噪声变大。
这部分通常对准确率提升最直接。

3. 再优化生成约束
如果检索结果已经比较准，但回答仍然不稳定，那我会看生成层。
比如：

在 prompt 里明确要求“只能基于给定材料回答”；
要求模型输出引用依据，降低幻觉；
对证据不足的问题，允许模型明确回答“不足以判断”；
对高风险场景做模板化或结构化输出，减少自由生成带来的偏差。
技术上这一步的目标不是让模型“更会说”，而是让它“更受控、更可靠”。

4. 最后回到数据和评测体系
还有一个很关键但经常被忽略的点，是数据和评测集本身。
如果训练/评测数据噪声很大，或者答案标准不一致，再怎么调模型也很难稳定提升。
所以我会：

清洗文档和问答数据；
补充高频失败场景的样本；
按问题类型拆分评测集，而不是只看一个总体准确率；
增加更细粒度的指标，比如召回率、MRR、命中率、答案可归因率。
因为只看一个总准确率，很难真正指导优化。

我会特别强调一点：
低准确率不可怕，可怕的是不知道低在哪里、也没有闭环去持续优化。
如果我已经建立了错误分析和迭代机制，那这个项目就仍然是往正确方向推进的。

Agent 的准确率评估，我一般不会只看单一 accuracy。
因为 Agent 是一个多阶段系统，不只是最后生成一句答案，它还包含任务规划、检索、工具调用和结果整合。

所以我通常分两层评估：
第一层是 结果层，看任务成功率和最终答案正确率；
第二层是 过程层，看检索召回、工具选择、参数调用、推理轨迹是否合理。

对于结构化任务，我会尽量做自动评估，比如字段匹配、数值校验、来源校验；
对于开放式任务，我会结合 LLM Judge 和人工复核。

除了准确率，我还会看稳定性、耗时、调用次数和成本，因为真正可用的 Agent，不只是答对，还要答得稳、答得快、答得可控。

我们实际会先构建一批标准任务集，每个任务包含问题、标准答案、关键证据或预期步骤。
跑完后先做自动比对，比如数值、实体、时间是否一致；
对自动评不了的样本，再用大模型评审或者人工复核。
最后把错误按检索错误、工具调用错误、推理错误、生成错误分类，这样后续优化会更有针对性。
